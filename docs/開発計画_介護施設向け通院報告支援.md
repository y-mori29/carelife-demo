# 介護施設向け通院報告支援システム — 開発計画書

- **作成日**: 2026年3月7日
- **前提**: 要件定義書 `20260307_要件定義書v01.md` に基づき、既存「よりそい」のソースを参照しつつ開発を進める

---

## 現状（2026年3月・実装済み）

- **Cloud Run**: バックエンド・フロントエンド・LINE Bot の 3 サービスをデプロイ済み。LINE Channel Access Token / Secret、Gemini API キー、Google Application Credentials を設定して本番稼働中。
- **音声〜報告生成**: 録音アップロード、GCS 保存、Speech-to-Text による文字起こし、Gemini による通院報告要約まで実装済み。
- **LINE 連携**: Bot が「報告」等に反応して報告作成リンク（`?userId=...` 付き）を返す。フロントの「LINEに送信する」で報告文を LINE Push 送信する機能まで実装済み。
- **補足入力**: 次回予約の日時・お迎え時間を含む補足項目を追加済み。録音終了後は補足入力画面へ直行する導線に変更済み。

上記を踏まえ、以下の「現状の整理」「フェーズ」は計画時点の記述として参照用に残しています。今後の拡張は「次にやること」および README の「今後の拡張」を参照してください。

### 1.1 ワークスペース構成

| パス | 内容 |
|------|------|
| （リポジトリのルート） | 本プロジェクトのルート（要件定義書・開発計画・backend / frontend / line-bot） |

| 既存「よりそい」のソース | 別リポジトリまたはローカルに用意する既存システム（患者向け・訪問薬局向け）の参照用 |

### 1.2 既存「よりそい」の構成（流用可能な部分）

| コンポーネント | 技術 | 主な役割 |
|----------------|------|----------|
| **backend** | Node.js (Express) | 録音受信、GCS保存、Speech-to-Text、Gemini による SOAP/薬剤抽出、Firestore 保存 |
| **frontend** | Next.js | 施設選択→患者選択→録音→送信、即時完了（非同期解析） |
| **dashboard** | Vite + React | 記録一覧・SOAP 閲覧・編集・音声再生 |

**流用しやすい要素**

- 録音エンジン（AudioWorklet、チャンク送信、高感度録音）
- バックエンドの音声正規化・STT（Speech-to-Text V2 / chirp_3）・Gemini 連携
- 非同期処理（finalize → 202 Accepted → バックグラウンドで解析）
- Firestore の `facilities` / `patients` / `encounters` の考え方（施設・患者・1回の記録）

**介護施設版で変える部分**

- **起動**: Web ログインではなく **LINE グループ内 Bot** から起動
- **画面フロー**: 要件の SC-01〜SC-12（起動→患者→担当者→録音→補足質問→確認→処理中→完了／エラー）
- **出力形式**: SOAP/薬剤ではなく **通院報告テンプレート**（利用者名・受診日・受診先・診察内容・ご本人の様子・待機中・申し送り・処方・送迎・次回予約・担当者名 など）
- **補足入力**: 自由記述中心ではなく **質問形式**（道中・待機中・申し送り・薬局・送迎 など）
- **連携**: 生成報告の **LINE 投稿** および **既存運用に沿った Chatwork 送信**

| 9.5 文字起こし | Speech-to-Text あり | 流用 |
| 9.6 補足入力 | なし（質問形式なし） | **新規**: 質問リスト定義と、SC-07〜SC-09 に相当する画面・API |
| 9.7 要約 | SOAP/薬剤向け | **拡張**: 通院報告用の要約プロンプトと出力スキーマを追加 |

| 9.9 LINE 投稿 | なし | **新規**: 報告文を LINE に投稿する API（Messaging API） |
| 9.10 Chatwork 連携 | なし | **新規**: 既存運用（LINE→Chatwork）に差し込む形で送信。API または手動コピー運用の整理 |
| 9.11 エラー案内 | 一部あり | SC-12 に合わせて文言・遷移を統一 |

| 画面ID | 要件の画面名 | 既存「よりそい」 | 対応 |
|--------|--------------|------------------|------|
| SC-01 | 起動画面 | ログイン画面 | LINE Bot 用の「開始する」入口に差し替え |

## 3. 開発の進め方（推奨フェーズ）

### フェーズ 0: 環境・方針の確定（短期） — 済

- 本番コードの置き場所: **リポジトリのルート直下に `backend` / `frontend` / `line-bot` を配置**（実施済み）。
- 認証・利用者識別: **MVP は LINE のみで完結**とする。
- 開発・検証用の LINE 公式アカウントと Bot、Firestore/GCP プロジェクトは**利用者側で用意し、後から連携**する。

### フェーズ 1: バックエンドの拡張（2〜3 週間想定）

1. **データモデル**
   - `encounters` に「通院報告」用フィールドを追加（または `carelife_reports` 等のコレクションを新設）。
   - 補足 Q&A の保存形式（質問 ID + 回答の配列）を定義。
2. **通院報告用 AI パイプライン**
   - 既存の `processingController` を拡張するか、`carelifeReportController` のような新モジュールを追加。
   - 入力: 文字起こし + 補足回答。
   - 出力: §10.2 の項目を満たす「通院報告」構造（JSON）。§10.3 のテンプレート文を生成。
3. **API 追加**
   - 補足質問一覧取得（GET）。
   - 録音 finalize 時に「補足回答」を受け取る（既存 finalize の拡張 or 新エンドポイント）。
   - 通院報告テキスト取得（生成完了後）。

ここまでで、既存フロントから「患者・施設・録音」を送り、補足を渡して通院報告が生成されることを確認できるようにする。

### フェーズ 2: フロント（画面フロー）の改修（2〜3 週間想定）

1. **画面 ID と遷移の整理**
   - SC-01〜SC-12 に対応する画面コンポーネントを定義（既存画面のリネーム・分割・新規追加）。
2. **フロー実装**
   - SC-01 → … → SC-06: 起動〜録音完了まで（既存の施設・患者・録音を SC-02〜SC-05 にマッピングし、SC-01/03/06 を追加）。
   - SC-07〜SC-09: 補足質問開始・回答・確認（新規）。
   - SC-10〜SC-11: 処理中表示、完了表示（LINE 送信済み案内）。
   - SC-12: エラー時共通画面。
3. **UX 要件の反映**
   - 1 画面 1 アクション、大きなボタン、短い文言、高齢者・非 IT 利用者配慮（§11.2〜11.3）。

この段階ではまだ「Web アプリ」として動作させ、LINE は経由しない形でもよい（後から LINE 経由に切り替え）。

### フェーズ 3: LINE 連携（1〜2 週間想定）

1. **LINE Bot**
   - LINE Messaging API で Webhook 受信。
   - 報告作成開始時に、録音・補足入力用の WebView（または LIFF）URL を返す。
2. **起動導線**
   - グループ内で Bot が「通院報告を作成します」「開始する」等のメッセージ＋URL を送る。
3. **認証・識別**
   - LINE の `userId` / `displayName` を担当者として保存（必要に応じて施設・役割と紐付け）。

### フェーズ 4: LINE 投稿・Chatwork 連携（1〜2 週間想定）

1. **LINE 投稿**
   - 報告生成完了後に、LINE Messaging API でグループ（または個人）に報告テキストを投稿する API を実装。
2. **Chatwork**
   - 既存運用（LINE で受け取った内容を Chatwork に転送）を維持する場合、  
     - オプション A: 報告文を LINE に投稿し、従来どおり人が Chatwork に転送。  
     - オプション B: バックエンドから Chatwork API で送信する機能を追加し、設定がある施設のみ自動送信。  
   要件では「既存運用に基づいて Chatwork へ送信」とあるため、まずはオプション A で受け入れ条件を満たし、必要なら B を検討。

### フェーズ 5: 受け入れ・チューニング（継続）

- §17 の受け入れ条件のチェックリストで検証。
- 補足質問の文言・数、出力テンプレートの微調整、エラー文言の改善。
- 画面モックや利用者テストを反映した UI 調整。

## 4. 技術的な注意点・決定事項

| 項目 | 内容 |
|------|------|
| **LINE と録音** | LINE アプリ内ではマイク直接利用に制約があるため、録音は **WebView / LIFF 内の Web アプリ** で行う構成が現実的。既存 frontend の録音部分を LIFF に組み込む形を推奨。 |
| **患者一覧** | 施設ごとの患者一覧は既存 `GET /api/patients` 等を流用。LINE 側では施設をどう選ぶか（1 Bot 1 施設 or 選択）を決める必要あり。 |
| **認証** | LINE ログインのみでよいか、施設の認証基盤と連携するかは要件・セキュリティ方針で決定。 |
| **Chatwork** | 既存運用が「手動で LINE から Chatwork にコピー」であれば、自動送信は「今後の検討事項」としてフェーズ 4 ではオプションにしてもよい。 |
